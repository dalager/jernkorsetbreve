# ADR-033: 3D Embedding Coordinates in Data Pipeline

## Status

Accepted (implemented 2026-03-28)

## Date

2026-03-28

## Context

The explorer page (`/explorer/`) visualizes 665 letters as a 2D scatter plot using UMAP-projected coordinates stored in `embeddings-2d.json`. The projection is generated by `scripts/generate-embeddings.mjs`, which runs UMAP with `nComponents: 2` on 384-dimensional embeddings from the `Xenova/multilingual-e5-small` model.

ADR-034 proposes adding an optional 3D explorer view using Three.js. That view needs 3D coordinates. The `umap-js` library already supports `nComponents: 3` natively -- the change is a small extension of the existing pipeline, not a new tool or model.

The question is how to produce and store the 3D coordinates alongside the existing 2D data.

## Decision

### 1. Add a second UMAP pass with `nComponents: 3`

In `scripts/generate-embeddings.mjs`, after the existing 2D UMAP projection (step 9), add a 3D projection:

```javascript
const umap3d = new UMAP({
  nNeighbors: 15,
  minDist: 0.1,
  nComponents: 3,
  spread: 1.0,
});

const projected3d = umap3d.fit(embeddingMatrix);
```

The same UMAP hyperparameters (`nNeighbors: 15`, `minDist: 0.1`, `spread: 1.0`) are used for both 2D and 3D projections to maintain consistency.

### 2. Output to `embeddings-3d.json`

Write the 3D coordinates to a new file `apps/website/public/data/embeddings-3d.json` with the same structure as the 2D file:

```json
{
  "method": "umap",
  "params": { "nNeighbors": 15, "minDist": 0.1, "nComponents": 3 },
  "points": [
    { "id": 1, "x": 0.4231, "y": 0.7182, "z": 0.3015 },
    ...
  ]
}
```

Coordinates are normalized to [0, 1] on all three axes, matching the 2D convention.

### 3. Include in cache invalidation

The existing cache check in `generate-embeddings.mjs` verifies that `embeddings-2d.json` exists. Add `embeddings-3d.json` to the same check so that both files are regenerated together when the corpus changes.

### 4. Keep 2D and 3D as separate files

The 2D file remains unchanged. The 3D file is loaded only when the user toggles to 3D view (see ADR-034), keeping the default 2D page load unaffected.

## Alternatives Considered

### Compute 3D coordinates client-side from the binary embeddings

The website already ships `embeddings.bin` (665 x 384 Float32 = ~1 MB). In theory, the client could run UMAP in the browser. Rejected because:
- UMAP on 665 x 384 vectors takes several seconds even on fast hardware
- `umap-js` would add to the main bundle unless carefully code-split
- The projection is deterministic for a given corpus; precomputing it avoids repeated work
- The binary embeddings file is served for search, not visualization -- coupling these concerns is undesirable

### Store 3D coordinates in the same file as 2D

Could add a `z` field to the existing `embeddings-2d.json` points. Rejected because:
- 2D and 3D are independent UMAP runs with different manifold structures; the x/y values differ between the two projections
- Combining them would force loading 3D data even when only 2D is needed
- Renaming `embeddings-2d.json` would break the existing consumer

### Use PCA or t-SNE instead of UMAP for the 3D projection

UMAP is already in use for 2D. Using the same algorithm for 3D preserves visual consistency (similar neighborhood relationships) and requires no new dependencies. t-SNE tends to produce more crowded clusters that work less well in 3D interactive exploration. PCA is linear and loses too much structure at 384 -> 3 dimensions.

## Consequences

### Positive

- 3D coordinates are available for the Three.js explorer (ADR-034) with no runtime computation cost
- Pipeline is self-consistent: both projections regenerate together on corpus change
- No new dependencies -- `umap-js` is already installed
- The 3D file is ~40 KB (665 points x 3 coordinates), negligible impact on build output

### Negative

- Pipeline runtime increases by the duration of one additional UMAP fit (~5-15 seconds for 665 points)
- One more output file to track in the data directory

## Implementation Notes

The change is approximately 30 lines in `scripts/generate-embeddings.mjs`:
1. A second UMAP instantiation with `nComponents: 3` (~5 lines)
2. Normalization of x/y/z to [0, 1] (~10 lines)
3. JSON serialization and file write (~5 lines)
4. Adding `embeddings-3d.json` to the cache existence check (~1 line)
5. Console logging for the 3D step (~5 lines)

Estimated effort: under 30 minutes.
