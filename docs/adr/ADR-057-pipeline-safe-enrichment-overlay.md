# ADR-057: Pipeline-Safe Enrichment Overlay

## Status

Accepted (2026-04-06)

## Date

2026-04-06

## Context

The NLP pipeline (`build-person-registry.py`) generates `data/person-registry.json` from NER data. The admin app and enrichment scripts add human-curated fields to the same file: `full_name`, `birth_date`, `death_date`, `biographical`, `photos`, `enrichment_source`, and Danish-translated `role` values.

Running `npm run data:person-registry` (or `data:network-all`, or `data:all`) **overwrites the entire file**, destroying all human enrichments and translations. This is a data-loss time bomb — someone or CI will eventually run the full pipeline.

The same problem exists for `places-enriched.json` (written by both `enrich-places-wikidata.py` and the admin app). `image-registry.json` is safe — no pipeline script writes to it.

### Severity

| File | Pipeline writes? | Admin writes? | Risk |
|---|---|---|---|
| `person-registry.json` | YES (full overwrite) | YES | **CRITICAL** |
| `places-enriched.json` | YES (full overwrite) | YES | **HIGH** |
| `image-registry.json` | NO (read-only) | YES | Safe |

## Decision

### Split-File Architecture with Merge-on-Build

Separate machine-generated data from human curation into distinct files. A merge step combines them, with explicit field ownership rules.

```
NLP Pipeline writes:         data/person-registry-computed.json
Human curation lives in:     data/person-registry-enrichments.json
Merge step produces:         data/person-registry.json  (consumed by website + admin)
```

### Field Ownership

| Owner | Fields | Rule |
|---|---|---|
| **Pipeline** (recomputed) | `letter_count`, `first_mention`, `last_mention` | Always taken from computed |
| **Human** (preserved) | `role`, `category`, `full_name`, `birth_date`, `death_date`, `biographical`, `photos`, `enrichment_source` | Human version wins if present; pipeline value as fallback |
| **Additive** | `aliases` | Union of pipeline aliases + enrichment `add_aliases` |
| **Human-only records** | Entire person entries with 0 NER mentions | Appended from enrichments `_manual_persons` list |

### Merge Rules (pseudocode)

```python
PIPELINE_OWNED = {"letter_count", "first_mention", "last_mention"}
HUMAN_OWNED = {"role", "category", "full_name", "birth_date", "death_date",
               "biographical", "photos", "enrichment_source"}

for person in computed:
    enrichment = enrichments.get(person.id)
    if enrichment:
        # Pipeline statistics always win
        for field in PIPELINE_OWNED:
            merged[field] = person[field]
        # Human curation wins if present
        for field in HUMAN_OWNED:
            if enrichment.has(field):
                merged[field] = enrichment[field]
            else:
                merged[field] = person.get(field)
        # Aliases: union
        merged.aliases = set(person.aliases) | set(enrichment.add_aliases)
    else:
        merged = person  # no enrichment, use pipeline output as-is

# Append manual-only persons not generated by pipeline
for manual_person in enrichments._manual_persons:
    if manual_person.id not in computed_ids:
        merged_list.append(manual_person)
```

### Pipeline Script Changes

1. `build-person-registry.py`: Change output from `person-registry.json` to `person-registry-computed.json` (one-line change)
2. New `merge-person-registry.py`: Loads computed + enrichments, applies merge rules, writes `person-registry.json`
3. `package.json`: Chain them: `"data:person-registry": "python scripts/build-person-registry.py && python scripts/merge-person-registry.py"`

### Admin App Changes

`savePersonRegistry()` in `apps/admin/src/lib/registry.ts` dual-writes:
1. `person-registry.json` (immediate consumption by website/admin)
2. `person-registry-enrichments.json` (survives next pipeline run)

### Places Pipeline

`enrich-places-wikidata.py` gets the same treatment:
- Output → `places-enriched-wikidata.json`
- Manual overrides → `places-enriched-overrides.json`
- Merge → `places-enriched.json`

### Safety Guard

`build-person-registry.py` refuses to run if `person-registry.json` contains enrichments but `person-registry-enrichments.json` doesn't exist (migration not yet complete).

## Consequences

### Positive
- `npm run data:all` becomes safe — no data loss risk
- Provenance is explicit: computed vs curated fields live in separate files
- NLP pipeline stays independently re-runnable
- No database needed — all JSON, all git-trackable
- Admin app enrichments survive pipeline re-runs indefinitely

### Negative
- One-time migration step required (extract enrichments from current file)
- Admin save is slightly more complex (dual-write)
- Three files instead of one for person data (but merge is automated)

### Neutral
- `image-registry.json` needs no changes (already safe)

## Depends On

- ADR-053 (atomic writes — enrichments file uses same pattern)
- ADR-051 (export — ZIP export should include enrichments file)

## Implementation Steps

1. `scripts/extract-person-enrichments.py` — one-time migration, split current file
2. `scripts/build-person-registry.py` — change output path (1 line)
3. `scripts/merge-person-registry.py` — new merge script (~80 lines)
4. `package.json` — chain build + merge in `data:person-registry` script
5. `apps/admin/src/lib/registry.ts` — add `syncEnrichments()` for dual-write
6. `scripts/enrich-places-wikidata.py` — same split pattern for places
7. Safety guard in `build-person-registry.py`
8. Update `docs/data-pipeline.md`
9. Update `apps/admin/src/routes/export.ts` — include enrichments in ZIP export
