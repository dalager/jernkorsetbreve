# ADR-058: Sanitize API Schema for External Consumption

## Status

Implemented (2026-04-19)

## Date

2026-04-19

## Context

The published `api-schema.json` at `https://jernkorset.dk/data/api-schema.json` serves as the OpenAPI 3.1 contract for external consumers and AI agents. An audit found 20+ descriptions that leak internal build pipeline details instead of describing data semantics.

Examples of problematic descriptions:

| Current description | Problem |
|---|---|
| "Built by build-data.mjs from letters.csv + normalized-letters.json" | Exposes build script names and source file paths |
| "Generated by generate-sentiments-cvp.py from normalized-sentences.json" | Names internal Python scripts and intermediate data files |
| "assigned by 01_cleanup.ipynb from data/letters.csv" | References Jupyter notebooks and repo-internal CSV files |
| "see docs/data-pipeline.md" | Points to a repo-only file that external consumers cannot access |
| "Editable source file — see ADR-045" | References internal ADRs irrelevant to API consumers |

These descriptions confuse external consumers (including AI agents performing automated discovery) by implying they should know about the build pipeline. The schema's `servers` URL correctly points to `https://jernkorset.dk/data/`, but the descriptions contradict this by referencing source files that only exist in the git repository.

Additionally, some descriptions reference intermediate data files (`normalized-sentences.json`, `letters.csv`, `places.geojson`) that are never published — they exist only as pipeline inputs.

## Decision

Strip all build-pipeline references from `api-schema.json` descriptions. The published schema is a public contract — it describes data shapes and semantics, not implementation details.

### What to remove

1. **Build script references**: All "Built by X.mjs", "Generated by X.py", "assigned by X.ipynb" sentences
2. **Source file references**: Mentions of `letters.csv`, `normalized-sentences.json`, `normalized-letters.json`, `places.geojson`, and other repo-internal files
3. **Documentation cross-references**: "see docs/data-pipeline.md", "see ADR-NNN" and similar pointers to repo-internal documents
4. **Editability notes**: "Editable source file" and similar notes about the data curation workflow

### What to keep

1. **Data semantics**: What each field represents and what values to expect
2. **Format descriptions**: Data types, enums, date formats, coordinate systems
3. **Relationship descriptions**: How entities relate (e.g., "person IDs reference entries in person-registry.json")
4. **Historical context**: Domain-relevant descriptions (e.g., "Letters from WW1 Danish soldiers")

### Example transformation

Before:
```
"description": "Per-letter NLP entity extractions. Built by build-data.mjs from letters.csv + normalized-letters.json. Each entry contains NER-tagged persons, places, and military units."
```

After:
```
"description": "Per-letter NLP entity extractions. Each entry contains NER-tagged persons, places, and military units."
```

### Alternatives considered

1. **Keep as-is** — Rejected. Leaks internal details, confuses AI agents performing schema-driven discovery, and creates a false impression that consumers need access to build scripts.

2. **Maintain two schemas (internal + external)** — Rejected. The duplication burden is not justified. Internal documentation belongs in `docs/data-pipeline.md` and code comments, not in the published API contract. Two schemas would inevitably drift.

3. **Strip pipeline refs, keep descriptions focused on data semantics** — Chosen. One schema, clean descriptions. Internal documentation stays in the repo where it belongs.

## Consequences

### Positive

- External consumers (human and AI) get clear, actionable descriptions focused on data semantics
- AI agents performing automated API discovery no longer encounter misleading references to inaccessible files
- Clean separation: published schema documents the API contract; `docs/data-pipeline.md` documents the build process

### Negative

- Developers editing `api-schema.json` must remember not to re-introduce pipeline references (mitigated by this ADR serving as the policy reference)
- One-time editorial pass required across all schema descriptions

### Neutral

- Internal pipeline documentation is unaffected — `docs/data-pipeline.md`, code comments, and ADRs continue to document how data is built
